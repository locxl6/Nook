## 一、Project Proposal 项目提出

**项目简介：**

Nook 是一个基于本地大语言模型的 AI 聊天桌面应用。项目通过 Ollama 部署本地大模型（如 Qwen2.5），使用 FastAPI 构建后端 REST API 服务，前端采用 Electron + React + TypeScript 构建跨平台桌面客户端。系统支持多会话管理、SSE 流式响应、Markdown 实时渲染、深色/浅色主题切换等功能，所有数据存储在本地 SQLite 数据库中，无需联网即可使用，保障用户数据隐私。

**项目需实现目标：**

1. 实现基于 Ollama 的本地大模型聊天功能，支持 SSE 流式输出
2. 实现多会话管理（创建、删除、切换、历史消息加载、标题修改、关键词搜索）
3. 实现 Markdown 和 LaTeX 数学公式的实时渲染
4. 实现深色/浅色主题切换与响应式布局
5. 构建基于 Electron 的跨平台桌面应用（Windows/macOS/Linux）
6. 所有数据本地存储，保护用户隐私，无需联网

## 三、Risks Evaluating and Mitigating 风险评估和规避

### Technical Risks 技术风险

（团队成员开发环境不统一，开发工具不一致，编码规范不一致）

1. 团队成员对 Electron 和 Ollama 缺乏经验，需要时间学习技术栈
2. SSE 流式传输在 Electron 渲染进程中的消费实现较为复杂

**Resolution 解决：**

组长提前编写技术指南文档（docs.nook.locxl.site），明确技术栈和分工；各成员在自己分支独立开发，通过 Git 进行版本管理和代码审批；前端使用 fetch + ReadableStream 消费 SSE 流，参考成熟方案实现。

### Management Risks 管理风险

（团队成员生病，事假，效率低下等）

1. 开发周期短（1周），时间压力大
2. 成员可能有其他课程安排冲突，影响开发进度

**Resolution 解决：**

明确分工，每人负责独立模块并行开发；使用 Git 分支管理（每人一分支，审批后合并到 dev 分支）；每日同步进度，组长负责协调和联调。

### Other Risks 其它风险

（系统崩溃，硬盘坏了等…）

1. 本地模型推理性能受硬件限制，低配设备响应较慢
2. 开发环境配置不一致（操作系统、Python 版本、Node 版本差异）

**Resolution 解决：**

默认使用轻量模型 qwen2.5:1.5b，CPU 也能运行；后端使用 Python venv 虚拟环境管理依赖，前端使用 package-lock.json 锁定版本；README 文档记录完整环境配置步骤。
